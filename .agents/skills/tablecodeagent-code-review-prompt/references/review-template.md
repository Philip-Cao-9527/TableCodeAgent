# TableCodeAgent Code Review Prompt 底稿

本文件是 `tablecodeagent-code-review-prompt` 的专属底稿。生成最终 prompt 时按用户任务裁剪，不保留模板说明、空占位符、旧项目路径或旧项目命令。

````text
你要在当前仓库 `d:\桌面\TableCodeAgent` 完成一次 code review。默认只输出审查报告和建议，不直接修改代码；除非用户明确要求修复，否则不要 apply patch。

【审查目标】

1. 项目类型：面向复杂表格任务的轻量级 Coding Agent。
2. 重点判断：是否存在 bug、行为回归、契约漂移、错误处理风险、测试缺口、文档误导、真实 API 与非 API 验证边界混淆，或会误导 TableCodeAgent 能力表述的问题。
3. 不做事项：默认不直接修复代码，不运行真实 API benchmark，除非用户明确要求并提供必要 env 边界。

【审查范围】

请先根据用户指定范围审查。范围可能来自当前分支 diff、指定 commit、PR diff、用户指定文件或目录。不要把范围外的风格偏好包装成问题；如果发现范围外存在阻塞风险，单独列为“范围外风险”，并说明为什么需要关注。

【必读文件】

审查前必须先阅读并在进度中明确已读：

- `.codex/AGENTS.md`
- `README.md`
- `docs/reproduce/tablecodeagent_architecture.md`
- `docs/reproduce/why_table_code_agent.md`
- 本轮审查范围相关文件、diff、测试、报告或日志

先运行 `rg --files .agents .codex docs` 或等价命令核对真实路径。

【外部资料检索规则】

默认优先基于本地代码、diff、测试、文档、日志和 trace 做审查。以下情况可以调用 `$web-search` 辅助定位问题或支撑评审结论：

1. 用户明确要求联网检索、查最新资料、查官方文档、查论文、查 benchmark 或查工程实践。
2. 本轮审查依赖近期可能变化的信息，例如 LLM API、OpenAI-compatible tool calling schema、Agent 框架、benchmark 规则、第三方库行为、数据集格式、论文方法或官方实现。
3. 本地代码与外部规范可能存在冲突，需要确认当前官方口径或主流实现。

如果用户明确要求不联网，不要调用 `$web-search`。如果启用 `$web-search`，必须优先查官方文档、论文、作者仓库、数据集主页或 benchmark 主页；社区文章只能作补充。外部资料只能用于帮助判断风险、定位问题和提出验证建议，最终 finding 仍必须落到本仓库的代码、diff、配置、文档、测试、日志或 trace 证据上。报告中要列出使用过的关键来源链接，并说明它们如何影响审查结论。

【风险地图生成要求】

不要直接套固定审查清单。开始审查前，必须先基于项目规则、审查目标、对比基准和真实改动建立本轮项目化风险地图，再从风险地图生成审查重点。

风险地图至少覆盖：

1. 复杂表格任务口径：是否把项目缩窄为普通数据分析 Agent，或把未实现能力写成当前能力。
2. Agent 工具暴露：是否错误公开 product workflow、helper-assisted oracle、`implementation_hints`、`allowed_project_helpers`、`solve_py_suggestion` 或 `build_*_report()`。
3. no-helper benchmark：真实 LLM Agent benchmark 是否仍保持 no-helper 口径，非 API 测试是否被误写成真实 Agent 能力。
4. JSON / Pydantic 契约：`answer.json`、tool schema、task `output_contract` 是否与 pytest / validator 实际读取路径对齐。
5. trace / validation 证据：是否记录足够字段证明执行、失败类型、`SKIP`、API 调用和校验结果。
6. 文档当前性：README、架构文档或报告是否把历史状态写成当前状态。
7. 真实 API 与非 API 验证边界：是否把未运行、env 缺失、网络失败或 API `SKIP` 写成已通过。

这些维度只是最低要求。你必须根据真实改动补充项目专属风险，不能把示例当成审查范围上限。

【证据规则】

1. 每条确认问题都必须给出文件路径、证据、影响、最小修复建议和验证建议。
2. 文件链接优先使用可跳转 Markdown 相对路径；能定位到行号时写到行号。
3. 证据必须来自代码、配置、文档、测试、日志、trace、diff 或实际命令输出。
4. 没有证据的问题不要写成 finding，可放入开放问题或建议核对。

【报告保存要求】

1. code review 报告必须保存为单独 Markdown 文档。
2. 如果用户指定路径，保存到用户指定路径。
3. 如果用户未指定路径，按项目习惯选择 `docs/reproduce/` 下合适文件名，并在执行前说明原因。
4. 报告文件名应体现审查主题和日期。
5. 报告内所有文件链接使用可跳转 Markdown 相对路径交叉引用。

【输出格式】

最终输出必须 findings first，按严重程度排序：

```markdown
**审查发现**

**本轮风险地图**

- 从真实项目和改动中提炼出的风险维度。

### `严重级别`：问题标题

位置：可跳转 Markdown 相对路径链接；能定位时写到行号。

证据：代码、测试、日志或文档证据。需要多条证据时用短段落或紧凑列表组织，不要堆成难读的大段。

影响：用户影响、数据影响、评测影响或维护风险。说明为什么这是问题，而不只是代码现象。

最小修复建议：可行修复方向、涉及文件和预期影响；如果存在多个方向，用 1 到 3 个短小标题或编号方向区分，不要在证据不足时强行拍板。

验证建议：应该补跑或新增的测试。

是否阻塞：是否阻塞合并 / 发布 / 交付。

**开放问题**

- 无法从当前材料确认的问题。

**测试缺口**

- 未覆盖场景或未运行验证。

**简短总结**

- 一句话总结。
```

严重级别参考：

- 阻塞：会导致功能不可用、数据错误、发布失败、权限 / 隐私 / 安全高风险，或真实 benchmark 口径被破坏。
- 高：主要路径回归、核心契约破坏、错误结果、helper 泄露或失败归因不可用。
- 中：边界场景错误、缺失关键验证、文档明显误导或可恢复的行为偏差。
- 低：局部可维护性风险、轻微文案问题或测试补强建议。

如果没有发现明确问题，也必须生成报告，写明“未发现阻塞项或明确缺陷”、已检查范围、剩余测试缺口和残余风险。不要为了凑 finding 把风格偏好写成 bug。

【注意】

1. 默认不修改代码。
2. 不要只总结改了什么，必须优先找风险和证据。
3. 不要把历史测试结果写成本轮验证结果。
4. 如果需要后续修复，给出最小修复建议和验证建议，等待用户确认。
````
