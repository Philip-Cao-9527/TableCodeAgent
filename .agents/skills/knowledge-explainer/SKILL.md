# knowledge-explainer

## 最高约束（绝对优先）
- 字数约束是本技能的最高优先级约束，高于文风、排版和示例数量。
- 第一部分 `可直接口述回答（快速复习总结）` 少于 1000 字，或第二部分 `详细原理讲解（通俗版）` 少于 3000 字，均视为**输出失败**。
- 只要任一部分未达字数门槛，必须继续补写，直到两部分同时达标后才允许输出最终结果。
- 禁止以“提纲化压缩、空洞重复、删减公式解释”等方式变相缩水。
- 详细原理讲解必须从最基础概念开始，按层递进，耐心细致展开；禁止跳过关键前置概念、禁止直接进入高级术语堆砌。
- 字数下限是最低门槛，不是目标上限；禁止“刚超过 1000/3000 字就停笔”。必须根据问题复杂度与讲解深度动态扩展内容，确保用户真正能听懂、复述和迁移。
- 在满足字数与讲解完整性的前提下，输出排版必须优先保证可读性：能分点时优先分点，能拆成短段或小节时不要堆成连续大段文字，避免用户阅读负担过重。
- 重点结论、核心公式含义、易混淆区别、记忆抓手与高频易错点，允许并建议使用 Markdown `**加粗**` 做最小必要强调；但禁止通篇大面积加粗，避免削弱重点。

## 技能定位
- 用于把八股问题转化为两层答案：第一层可直接口述给面试官，第二层从零开始教学讲解。
- 面向“既要专业表达，又要通俗理解”的复习场景，默认支持公式与案例比喻并行输出。
- 当问题与 TableCodeAgent 项目强相关时，自动结合项目背景生成场景化回答；当问题为通用八股时，不强行绑定项目背景。
- TableCodeAgent 是为了算法/Agent/CodeAgent/数据分析 Agent 实习准备的项目，回答时优先映射到“表格任务、工具调用、代码执行、答案校验、trace、benchmark、失败分析”等真实项目链路。

## 何时触发
- 只要用户明确提出“八股讲解”“原理讲解”“面试八股复习”等需求，即触发本技能。
- 用户强调“用于面试口述”“需要专业但可记忆”“希望从零开始讲”。
- 用户要求输出包含公式、案例、比喻、记忆点。
- 用户要求解释 Agent Loop、tool use、context compression、memory、benchmark、trace logger、表格工具、验证指标等 TableCodeAgent 相关概念。
- 用户要求把 TableCodeAgent 的实现转成面试可讲的技术表达或项目追问回答。

## 何时不触发
- 仅要求一句话定义或极短摘要。
- 任务目标是直接写代码、调试报错、修改工程配置。
- 用户只要求学术文风或只要论文式推导，不需要口述版。

## 执行流程 / 使用步骤
1. 先识别问题类型：基础概念类、模型原理类、训练优化类、损失函数类、推理部署类、Agent 架构类、项目追问类。
2. 判断是否“TableCodeAgent 项目相关”：
- 若用户问题来自 TableCodeAgent 项目，先读取最小项目上下文：`README.md`、`docs/reproduce/2026-06-03_tablecodeagent_architecture.md`；必要时再读取 `python/mini_claude/agent.py`、`python/mini_claude/tools.py`、`python/tablecodeagent/table_tools/core.py`。
- 若用户问题是通用八股，不绑定项目背景，按通用教学结构输出。
3. 若问题涉及外部论文依据、最新方法对比、官方文档、数据集、benchmark 或争议结论，先调用 `$web-search` 再输出结论。
4. 组织回答顺序：
- 先写可直接口述版（专业、分点、可记忆）。
- 再写详细教学版（从零概念开始，不预设基础）。
- 若是项目追问，最后补充“面试官可能追问与回答”。
5. 所有公式统一使用 LaTeX 表达；复杂概念配通俗比喻与小案例。
6. 自检字数与结构，未达标不输出。
7. 输出时不要把未实现功能说成已完成。必须区分：已实现、当前 MVP 正在做、后续扩展方向。

## 输出要求 / 输出结构
1. `可直接口述回答（快速复习总结，>=1000字）`
- 必须分点（推荐 5-8 点）。
- 语气专业，适合面试直说。
- 必须包含关键公式（LaTeX）。
- 必须包含至少 1 个“记忆点/比喻/微案例”用于快速复习。
- 若问题与 TableCodeAgent 相关，每个核心点都要尽量回答“为什么重要”和“我在项目里怎么体现”。
- 不要夸大：没有 trace、benchmark、SFT/RL 实验时不能说已经做了。

2. `详细原理讲解（通俗版，>=3000字，含公式）`
- 必须从 0 开始讲，不预设用户懂机器学习/深度学习/Agent。
- 通用八股必须按“基础概念 -> 过渡模型 -> 目标模型 -> 训练与评估 -> 场景案例”顺序展开。
- Agent / TableCodeAgent 问题必须按“要解决什么问题 -> 普通做法不足 -> 核心机制 -> 项目代码落点 -> 训练/评测或工程闭环 -> 场景案例”顺序展开。
- 必须细讲关键机制（例如门控、注意力、损失分解、工具调用、上下文压缩、轨迹记录、答案校验等），不能草率略过。
- 必须给出公式解释、符号解释、直观意义解释。
- 必须至少包含 2 个通俗比喻或案例。
- 若涉及 TableCodeAgent，优先映射到 CLI 入口、Agent Loop、工具注册、表格工具、代码执行、答案校验、trace logger、benchmark runner、失败类型分析。

3. `面试官可能追问与回答（项目相关问题必须包含）`
- 至少给出 3-5 个面试官可能追问的问题，并给出简洁回答。
- 追问必须贴近项目真实状态，例如：
  - 为什么不是直接让 LLM 读 CSV 回答？
  - 表格工具和 `run_shell` 有什么区别？
  - 你怎么验证 Agent 算对了？
  - 当前 MVP 有哪些限制？
  - 后续为什么可以扩展到 WikiTQ / TabMWP？

## 质量门槛 / 注意事项
- 两部分字数为硬约束且为最高层约束：第一部分不少于 1000 字，第二部分不少于 3000 字；任一不达标即判定输出失败，必须继续补写后再交付。
- 第一部分偏“专业口述”，第二部分偏“新手教学”；两部分语气不得混淆。
- 公式不能只堆砌，必须配文字解释“每个符号代表什么、为什么这样写、解决什么问题”。
- 讲解必须可复述、可理解、可迁移，不得只给术语清单。
- 若问题涉及多个核心机制（如结构、训练、时序、对比、落地），必须显著高于最低字数下限，确保每个机制都有完整解释、示例和记忆点。
- 若用户要求“结合项目”，必须显式映射到 TableCodeAgent 项目链路；若用户未要求或属于通用题，不强行映射。
- 解释要耐心，适合代码小白理解，但不能牺牲技术准确性。
- 必须把“术语”翻译成“项目里发生了什么”。
- 不得机械生成简历 bullet；没有代码、实验、trace 或 README 证据时，只能说“后续可写”，不能说“已经完成”。
- 不要为了显得高级强行引入 RL、SFT、RAG、MCP、Memory；只有和问题强相关才解释。
- 若回答需要外部依据，先调用 `$web-search`，并在最终答案中说明依据来源。
