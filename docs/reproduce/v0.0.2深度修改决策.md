# v0.0.2 benchmark 口径重构决策

## 1. 结论

本项目后续只把真实 API 参与的端到端 Agent 能力评测称为 benchmark。非 API 检查仍然有价值，但应迁移到 `tests/`，作为单元测试、集成测试或 smoke / regression 入口，不再写成真实 Agent benchmark 成果。

本轮落地后的口径是：

- `real_api_code_agent` 是唯一核心真实 API benchmark。
- `benchmark_runner.py` 是真实 benchmark 总调度器。
- `src/tablecodeagent/benchmark/runner.py` 已删除，不保留 wrapper，不保留 deprecated 旧入口。
- 新 benchmark 结果只写入 `benchmarks/results/`。
- 历史 `benchmarks/runs/...` 只作为旧报告证据保留，不作为新路径模板。

## 3. 当前真实 benchmark 在测什么

`real_api_code_agent` 的最小闭环是：

1. 读取 `benchmarks/tasks/<task_id>/task.json` 和数据文件。
2. 创建 `benchmarks/results/real_api_code_agent/<run_id>/`。
3. 创建 workspace，并在模型生成与执行阶段隐藏 `expected.json`。
4. 调用真实 LLM API。
5. 要求模型生成 `solve.py`。
6. 保存模型生成代码，记录 `code_generation_source=llm_generated`。
7. 在 sandbox 中执行 `solve.py`，生成 `answer.json`。
8. 执行外部 pytest 或 validator，对照 `expected.json` 校验。
9. 写出 `results.jsonl`、`traces/`、`workspaces/`、`summary.json`。
10. API、依赖、env 或网络缺失时记录 `skipped=true` 和明确 `failure_type`。

入口文件：

- [benchmark_runner.py](../../src/tablecodeagent/benchmark/benchmark_runner.py)
- [real_api_code_agent.py](../../src/tablecodeagent/benchmark/real_api_code_agent.py)
- [logger.py](../../src/tablecodeagent/tracing/logger.py)
- [dependency.py](../../src/tablecodeagent/runtime/dependency.py)
- [sandbox.py](../../src/tablecodeagent/runtime/sandbox.py)

推荐命令：

```bash
python -m tablecodeagent.benchmark.benchmark_runner --env configs/api/local/provider_chatanywhere.env --task-dir benchmarks/tasks/growth_campaign_audit_001
```

或：

```bash
bash scripts/run_real_api_code_agent_benchmark.sh configs/api/local/provider_chatanywhere.env benchmarks/tasks/growth_campaign_audit_001
```

## 4. 为什么非 API 检查仍然保留

非 API 检查不是虚假的，但它们只能证明项目代码没有坏，不能证明 Agent 能力。

- `query_table` / `validate_answer` 测试用于定位底层表格函数错误。
- 工具 routing 测试用于定位 schema 注册、字段名、JSON 回包和 `execute_tool()` 接线错误。
- growth 质量函数测试用于定位领域规则错误。
- 固定 workflow 测试用于定位确定性审计输出结构错误。
- 固定 `solve.py` sandbox 测试用于定位代码落盘、进程执行、pytest 与 `answer.json` 闭环错误。

这些测试必须在 `tests/` 下运行，不写入 `benchmarks/results/`，也不应在 README 或 fix-report 中写成真实 API benchmark 成果。

## 5. 结果路径与证据字段

新结果目录命名规则：

```text
benchmarks/results/<benchmark_mode>/<YYYYMMDD-HHMMSS>__model-<model_name>__tasks-<task_group_or_task_id>/
```

示例：

```text
benchmarks/results/real_api_code_agent/20260606-035402__model-deepseek-v4-flash__tasks-growth_campaign_audit_001/
```

每个结果目录至少包含：

- `results.jsonl`
- `traces/`
- `workspaces/`
- `summary.json`

fix-report 必须记录：

- `result_dir`
- `trace_path`
- `workspace_path`
- `generated_code_path`
- `answer_path`
- `api_called`
- `skipped`
- `failure_type`
- `code_generation_source`
- `validation.passed`

## 6. pytest 缺失问题的处理

历史 `v0.0.2` 验证中，`pytest` 缺失会阻断 L2 sandbox 或后续真实 API demo 串行验证。这个问题不能写成通过，也不能被包装成“无数据”。

当前处理口径：

- `pytest` 属于 `TEST_DEPENDENCIES`。
- 真实 API benchmark 启动时会通过 `include_test=True` 纳入 dependency preflight。
- 缺依赖或安装失败必须记录 `dependency_missing` 或 `dependency_install_failed`。
- 真实 API benchmark 缺 env 时必须记录 `skipped=true` 和 `failure_type=api_env_missing`。

## 7. 对旧报告的解释

[fix-report-v0.0.2-20260606.md](./fix-report-v0.0.2-20260606.md) 中的 `benchmarks/runs/...` 是历史路径，不代表新 benchmark 输出规范。

旧报告里的 `sandbox_code_agent` 是固定 `solve.py` 基础设施测试，不是 LLM 自主生成代码 benchmark。

旧报告里的 `optional_llm_agent` 调用了真实 API，但 prompt 要求不修改文件，因此没有生成 `solve.py`。它只能证明真实 API tool-use 路径曾经跑通，不能证明真实 API 代码生成、sandbox 执行和 pytest 校验能力。
